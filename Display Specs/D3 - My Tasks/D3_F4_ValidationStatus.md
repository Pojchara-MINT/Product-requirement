# D3 — F4: Validation Status Display

> **Display:** D3 | **Function:** 4 of 4
> **Back to overview:** [D3_Overview.md](D3_Overview.md)

---

**Objective:** แสดงผลการตรวจสอบทั้งหมดใน Panel 3 ให้ Checker / Signer ประกอบการตัดสินใจ

| ส่วน | ชื่อ | ประมวลผลเมื่อ | ใครเห็น |
|------|-----|-------------|---------|
| 1 | ผลตรวจสอบบัญชีปลายทาง | FTT เข้าสู่ `PENDING_CHECKER` / `PENDING_SIGNER` | Checker, Signer |
| 2 | ผลเงื่อนไขโอนเงิน | Real-time | Checker, Signer |
| 3 | Approval-time Validation | กดปุ่ม ยืนยัน | Checker, Signer |
| 4 | ผลการโอนเงินจากธนาคาร | หลัง Signer กดยืนยัน → ธนาคารตอบกลับ | Signer เท่านั้น |

> ส่วนที่ 1 และ 2 — ผู้ตรวจ**อ่านได้อย่างเดียว** ไม่สามารถแก้ไขได้

---

## ส่วนที่ 1 — ผลตรวจสอบบัญชีปลายทาง

ผลการ Verify บัญชีปลายทางกับธนาคาร (Bank Step 1)

### Display Layout

| Element | Description |
|---------|-------------|
| Section Header | **ผลตรวจสอบบัญชีปลายทาง** — แสดง Accordion (ขยาย/ย่อได้) |
| Status Badge (Header) | ผ่าน (เขียว) / ไม่ผ่าน (แดง) / ยังไม่ได้รับผล (เทา) |

### เมื่อผลตรวจสอบบัญชีปลายทาง = ไม่ผ่าน

แต่ละ Error จะแสดงเป็น Sub-section พร้อมข้อมูลดังนี้:

| Field | Description |
|-------|-------------|
| ชื่อ Error | ข้อความอธิบายเหตุผล (เช่น "บัญชีไม่ตรงกับข้อมูลที่ลงทะเบียนในระบบธนาคาร") |
| เลขบัญชีที่อ้างอิง | เลขบัญชีที่ส่งไปตรวจ |
| ธนาคารปลายทาง | ชื่อธนาคาร |
| รหัสตอบกลับ | Response code จากธนาคาร |

### Bank Response Codes

| รหัสตอบกลับ | Display Text (ชื่อ Error) | Retryable |
|------------|--------------------------|:---------:|
| 4000 | บัญชีไม่ตรงกับข้อมูลที่ลงทะเบียนในระบบธนาคาร | TBD |
| 4001 | ไม่พบข้อมูลบัญชีปลายทาง | TBD |
| ... | TBD — รอข้อมูลเพิ่มเติมจาก Dev / ธนาคาร | TBD |

> **Retryable = ใช่** → ระบบแสดงปุ่ม **Retry** แทนปุ่ม ยืนยัน ใน Panel 3
> **Retryable = ไม่ใช่** → ระบบแสดงปุ่ม **ยืนยัน** ตามปกติ (User ตัดสินใจ Approve หรือ Reject เอง)

## ส่วนที่ 2 — ผลเงื่อนไขโอนเงิน

ผลการตรวจสอบ Internal Business Rule ของระบบ

### Display Layout

| Element | Description |
|---------|-------------|
| Section Header | **ผลเงื่อนไขโอนเงิน** — แสดง Accordion (ขยาย/ย่อได้) |
| Status Badge (Header) | ผ่าน (เขียว) / ไม่ผ่าน (แดง) / ยังไม่ได้รับผล (เทา) |

### เมื่อผลเงื่อนไขโอนเงิน = ไม่ผ่าน

แต่ละ Rule Violation จะแสดงเป็น Sub-section โดยระบุ FTT ที่ขัดแย้ง:

| Field | Description |
|-------|-------------|
| รหัสรายการ | FTT ID ที่ทำให้เกิด Rule Violation |
| หลักประกันที่ซ้ำ | หลักประกันที่ซ้ำกับรายการนี้ |
| เลขที่บัญชีที่ซ้ำ | เลขบัญชีที่ซ้ำกับรายการนี้ |
| ประเภทญัตติที่ซ้ำ | ประเภทสินเชื่อที่ซ้ำ |

### Internal Business Rules

| 2 | รหัสบัตรประชาชน + ประเภทสินเชื่อ + ประเภทสัญญา | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |
| 3 | รหัสบัตรประชาชน + ยอดเงินโอนสุทธิ | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |
| 4 | เลขที่บัญชี | ซ้ำ ≥ 3 สัญญาในวันเดียวกัน | — |
| 5 | เลขที่บัญชี + ประเภทสินเชื่อ + ประเภทสัญญา | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |
| 6 | เลขที่บัญชี + ยอดเงินโอนสุทธิ | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |
| 7 | หลักประกัน | ซ้ำ > 1 สัญญาภายใน 1 วัน | ไม่ตรวจนาโน |
| 8 | หลักประกัน + ประเภทสินเชื่อ + ประเภทสัญญา | ซ้ำ > 1 สัญญาภายใน 7 วัน | ไม่ตรวจนาโน |
| 9 | หลักประกัน + ประเภทสินเชื่อ + ยอดเงินโอนสุทธิ | ซ้ำ > 1 สัญญาภายใน 7 วัน | ไม่ตรวจนาโน |
| 10 | เลขที่พร้อมเพย์ | ซ้ำ ≥ 3 สัญญาในวันเดียวกัน | — |
| 11 | เลขที่พร้อมเพย์ + ประเภทสินเชื่อ + ประเภทสัญญา | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |
| 12 | เลขที่พร้อมเพย์ + ยอดเงินโอนสุทธิ | ซ้ำ > 1 สัญญาภายใน 7 วัน | — |

> **หมายเหตุ:** Rule 7, 8, 9 — ไม่ตรวจสอบสำหรับประเภทสินเชื่อนาโน

## ส่วนที่ 3 — การตรวจสอบ ณ เวลาอนุมัติ (Approval-time Validation)

> ตรวจสอบเมื่อ User กดปุ่ม **ยืนยัน (Approve)** เท่านั้น — ไม่แสดงใน Panel 3 ล่วงหน้า

### Rules

| # | เงื่อนไขที่ตรวจ | ใช้กับ | Error Message |
|---|--------------|-------|--------------|
| 1 | สถานะบัญชีสินเชื่อเปิดอยู่หรือไม่ | Direct Charge Application | สถานะสัญญานี้ปิดแล้ว |
| 2 | สถานะบัญชีสินเชื่อต้นทางเปิดอยู่หรือไม่ | Top Up / Restructure Application | สถานะสัญญาต้นทางปิดแล้ว |

### Flow เมื่อตรวจไม่ผ่าน

| Step | Action | System Response |
|------|--------|----------------|
| 1 | User กดปุ่ม **ยืนยัน** | ระบบตรวจสอบสถานะบัญชีสินเชื่อ |
| 2 | ตรวจพบสถานะปิด | ระบบแสดง **Error Modal** พร้อมข้อความ Error |
| 3 | User ปิด Modal | **ปุ่ม ยืนยัน หายออกจาก Panel 3** — เหลือเฉพาะปุ่ม ยกเลิก |
| 4 | User กดปุ่ม **ยกเลิก** | FTT ถูก Reject ตาม Reject Flow ปกติ |

> เมื่อ Approval-time Validation ไม่ผ่าน → User **ไม่สามารถ Approve** รายการนี้ได้อีก — ทำได้เฉพาะ Reject เท่านั้น

## ส่วนที่ 4 — ผลการโอนเงินจากธนาคาร

> **แสดงเฉพาะ Signer** — ปรากฏหลังจาก Signer กดยืนยัน แล้วธนาคารตอบกลับผลการโอนเงิน

### Trigger

| Step | Event |
|------|-------|
| 1 | Signer กดปุ่ม **ยืนยัน** |
| 2 | ระบบส่งคำสั่งโอนเงินไปธนาคาร (Bank Step 2) |
| 3 | ธนาคารตอบกลับผลการโอนเงิน |
| 4 | ระบบแสดงผลใน Section **ผลการโอนเงินจากธนาคาร** ใน Panel 3 |

### Use Cases

**Case 1: Transfer สำเร็จ**

| Step | Result |
|------|--------|
| ธนาคารตอบ: สำเร็จ | FTT State → `BANK_SUBMITTED` |
| | FTT หายออกจาก D3 ของ Signer |
| | ไม่แสดง Section ผลการโอนเงินจากธนาคาร |

**Case 2: Transfer ล้มเหลว**

| Step | Result |
|------|--------|
| ธนาคารตอบ: ล้มเหลว | FTT State คงอยู่ที่ `PENDING_SIGNER` |
| | แสดง Section **ผลการโอนเงินจากธนาคาร** (สีแดง) ใน Panel 3 |
| | แสดงรายละเอียด error และ Response Code |
| | Signer ยังสามารถกด **ยืนยัน** (retry) หรือ **ยกเลิก** ได้ |

### Display Layout (กรณี Transfer ล้มเหลว)

| Element | Description |
|---------|-------------|
| Section Header | **ผลการโอนเงินจากธนาคาร** — Accordion (แดง) |
| Sub-section | ชื่อ Error (เช่น "จำนวนเงินในบัญชีไม่เพียงพอ") |
| เลขบัญชีที่อ้างอิง | เลขบัญชีที่ส่งโอน |
| ธนาคารปลายทาง | ชื่อธนาคาร |
| รหัสตอบกลับ | Response Code จากธนาคาร |

## สถานะงาน vs ผลการตรวจสอบ

| ผลตรวจสอบบัญชีปลายทาง | ผลเงื่อนไขโอนเงิน | สถานะงาน |
|----------------------|-----------------|---------|
| ได้รับผลแล้ว (ผ่าน หรือ ไม่ผ่าน) | ได้รับผลแล้ว (ผ่าน หรือ ไม่ผ่าน) | พร้อมตรวจ (เขียว) |
| ยังไม่ได้รับผล | — (ไม่ว่าอีกรายการจะเป็นอะไร) | ไม่พร้อมตรวจ (ส้ม) |
| — (ไม่ว่าอีกรายการจะเป็นอะไร) | ยังไม่ได้รับผล | ไม่พร้อมตรวจ (ส้ม) |

> **หมายเหตุ:** User สามารถ Approve / Reject ได้แม้ว่าสถานะงาน = ไม่พร้อมตรวจ — ขึ้นอยู่กับการตัดสินใจของ Checker / Signer
